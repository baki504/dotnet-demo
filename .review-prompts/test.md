# セルフレビュー観点: テストコード

## テスト設計
- [ ] テストメソッド名が「何を」「どの条件で」「どうなるか」を表しているか（例: CreateEmployee_WithDuplicateEmail_ThrowsException）
- [ ] 1テストメソッドにつき1つのアサーションか（関連する複数アサーションは許容）
- [ ] Arrange / Act / Assert（AAA）パターンに従っているか
- [ ] テストが他のテストに依存していないか（独立して実行可能か）
- [ ] テストの実行順序に依存していないか
- [ ] 同一メソッドに対する複数の入力パターンテストは [Theory] + [InlineData] を使ってパラメータ化しているか（[Fact] の羅列で済ませていないか）

## テストケースの並び順
- [ ] テストメソッドが意味のあるまとまりで並んでいるか（正常系 → 異常系 → 境界値 → エッジケースの順）
- [ ] [Theory] の [InlineData] は意味のまとまりごとに並んでいるか（例: null/空 → 正常値 → 境界値 → 異常値、または「なし → あり」の順）
- [ ] まとまりの境目に空行やコメントを入れて可読性を確保しているか

## テストの網羅性
- [ ] 正常系（Happy Path）がテストされているか
- [ ] 異常系（バリデーションエラー、存在しないリソース、権限不足）がテストされているか
- [ ] 境界値がテストされているか（最小値、最大値、空、null）
- [ ] ビジネスルールの各条件分岐がテストされているか
- [ ] エッジケースが考慮されているか

## テストの品質
- [ ] テストがリファクタリングに耐えるか（実装の詳細でなく振る舞いをテスト）
- [ ] テストデータが明示的で読みやすいか（マジックナンバーを避ける）
- [ ] テストヘルパー / ファクトリメソッドで重複するセットアップを共通化しているか
- [ ] テストが高速に実行できるか（外部依存なし、I/O なし）

## モック
- [ ] モックが必要最小限か（テスト対象の外部依存のみ）
- [ ] モックの Setup が過度に複雑になっていないか（複雑ならテスト設計を見直す）
- [ ] モックの Verify が適切か（呼ばれるべきメソッドが呼ばれたか）
- [ ] モックがテスト対象の内部実装に強く依存していないか

## xUnit 固有
- [ ] [Fact] と [Theory] が適切に使い分けられているか（パラメータ化テストは [Theory] + [InlineData]）
- [ ] IClassFixture / ICollectionFixture を使った共有コンテキストが適切か
- [ ] Assert メソッドが適切か（Assert.Equal, Assert.Throws, Assert.NotNull 等）
- [ ] 非同期テストが async Task を返しているか（async void ではない）
