# セルフレビュー観点: 共通

## 命名規則
- [ ] PascalCase: public メンバー、クラス、メソッド、プロパティ
- [ ] _camelCase: private フィールド
- [ ] camelCase: ローカル変数、パラメータ
- [ ] 名前が役割を正確に表しているか（略語や曖昧な名前を避ける）
- [ ] bool 型は Is/Has/Can/Should で始まるか

## 設計原則（SOLID）
- [ ] SRP: クラスに変更理由が1つだけか
- [ ] OCP: 拡張に開き、修正に閉じているか
- [ ] LSP: 派生クラスが基底クラスの契約を守っているか
- [ ] ISP: インターフェースが不要なメソッドを強制していないか
- [ ] DIP: 具象ではなく抽象に依存しているか

## コード品質
- [ ] DRY: ロジックの重複がないか
- [ ] メソッドが40行を超えていないか（超える場合は分割を検討）
- [ ] ネストが3段を超えていないか（早期リターンやガード句で平坦化）
- [ ] マジックナンバー・マジックストリングが定数化されているか
- [ ] 不要なコメントがないか（コードが自己文書化されているか）
- [ ] 必要なコメントがあるか（WHYを説明するコメント）
- [ ] 未使用の using / 変数 / メソッドが残っていないか

## エラーハンドリング
- [ ] 例外を正常フローの制御に使っていないか
- [ ] catch で例外を握りつぶしていないか（少なくともログ出力）
- [ ] エラーメッセージがデバッグに有用かつ機密情報を含まないか
- [ ] IDisposable リソースは using で確実に解放しているか

## セキュリティ
- [ ] API キー・接続文字列等がハードコードされていないか
- [ ] 入力値の型・長さ・形式をバリデーションしているか
- [ ] インジェクション対策がされているか（具体的なチェックはレイヤー別の観点を参照）

## 非同期処理
- [ ] async void を使っていないか（イベントハンドラ以外）
- [ ] await の漏れがないか（Task を返すメソッドの呼び出し）
- [ ] ConfigureAwait が必要な箇所で使われているか（ライブラリコードの場合）
